Claude Code CLI – Kompletní tutoriál

Průvodce od instalace po pokročilé multi-agent architektury

ℹ️

Instalace

Potřebujete Node.js 18+ a API klíč od Anthropic

Požadavky

  • Node.js verze 18 nebo novější (node --version)
  • npm (součást Node.js)
  • Anthropic API klíč nebo přihlášení přes Claude.ai Pro/Max

Globální instalace

$ npm install -g @anthropic-ai/claude-code
⚠️

Pokud dostanete chybu oprávnění na macOS/Linux, nepoužívejte sudo. Místo toho nastavte správně prefix npm: npm config set prefix '~/.npm-global' a přidejte ~/.npm-global/bin do PATH.

Přihlášení

Spusťte Claude Code

$ claude

Zvolte způsob přihlášení

Claude Code vám nabídne tři možnosti:

  • Claude.ai – přihlásíte se přes prohlížeč (Pro/Max plán)
  • Anthropic API klíč – zadáte klíč přímo
  • Bedrock / Vertex – cloudové deployments

Ověření instalace

$ claude --version
@anthropic-ai/claude-code 1.x.x

$ claude doctor
✓ CLI nainstalováno
✓ Přihlášen jako user@example.com
✓ Připojení k API OK

První kroky

Jak efektivně komunikovat s Claude Code od prvního dne

Interaktivní REPL

Spusťte claude v kořenovém adresáři vašeho projektu. Claude automaticky přečte váš projekt, git historii a pochopí kontext.

$ cd ~/projects/muj-projekt
$ claude
╭────────────────────────────────────────╮
│  Claude Code  (Sonnet 4.6)             │
│  Pracovní adresář: ~/projects/muj-projekt │
╰────────────────────────────────────────╯

Vy: Analyzuj strukturu projektu a řekni mi, co dělá

Claude: Podívám se na strukturu projektu...
[čte soubory, analyzuje kód]
Toto je Flask aplikace se třemi moduly...

Typy požadavků

🔍

Analýza

"Proč tento test selhává?" / "Vysvětli mi tuto funkci"

🛠️

Oprava chyb

"Oprav bug na řádku 45 v auth.py"

✨

Nová funkce

"Přidej endpoint pro reset hesla"

♻️

Refaktoring

"Přepište tuto třídu na funkce"

Jak psát dobré prompty

💡

Buďte konkrétní. Místo „oprav kód" napište „oprav funkci calculate_tax() v souboru billing.py – vrací záporné hodnoty pro vstupy nad 1000".

Špatný promptDobrý prompt
„Udělej to lepší"„Optimalizuj SQL dotaz v db.py:45 – trvá 3s na 10k záznamech"
„Přidej přihlašování"„Přidej JWT autentizaci do Flask API, tokeny platné 24h"
„Proč to nefunguje?"„Test test_user_creation selže s IntegrityError, zde je traceback: …"

Schvalování akcí

Claude Code vás vždy požádá o souhlas před:

  • Spuštěním shell příkazů (Bash)
  • Zápisem nebo smazáním souborů
  • Pushováním do git remote
  • Voláním externích API

Odpovídáte y (ano) nebo n (ne). Pro automatické schvalování určitých příkazů použijte allowedTools v settings.json.

Kontextové okno a /compact

Při dlouhých sezeních se kontext zaplní. Claude vás varuje a nabídne /compact – automaticky shrne historii a uvolní místo pro nové úkoly bez ztráty důležitého kontextu.

Příklad: Bash nástroj

# Claude navrhne spuštění:

    "deny": [
      "Bash(rm -rf *)"
    ]
  }

CLAUDE.md

Soubor s instrukcemi, který Claude automaticky čte na začátku každého sezení

Co je CLAUDE.md?

CLAUDE.md je Markdown soubor, který Claude Code načte automaticky. Slouží jako „systémový prompt" specifický pro váš projekt – popisuje konvence, zákazy, architekturu a cokoliv, co má Claude vždy vědět.

Umístění souborů

CestaPlatnostPoužití
~/.claude/CLAUDE.mdGlobálníVaše osobní preference platné ve všech projektech
./CLAUDE.mdProjektPravidla konkrétního projektu (commitujte do gitu)
src/CLAUDE.mdPodadresářSpecifická pravidla pro část projektu

Generování pomocí /init

$ claude
Vy: /init

Claude prozkoumá váš projekt a automaticky vytvoří relevantní CLAUDE.md s popisem architektury, testovacích příkazů a konvencí.

Šablona CLAUDE.md

# CLAUDE.md – Moje Flask aplikace

## Projekt
REST API pro správu skladu. Backend Python/Flask, databáze PostgreSQL,
testy pytest, deployment na AWS.

## Příkazy
- Spuštění: `flask run --debug`
- Testy: `pytest tests/ -v`
- Migrace: `flask db migrate && flask db upgrade`
- Linter: `ruff check . && mypy src/`

## Konvence kódu
- Python 3.12, PEP 8, type hints vždy
- Funkce místo tříd kde je to možné
- Žádné `print()` – pouze `logger.info()`
- Všechny API endpointy v `src/api/`, modely v `src/models/`

## Zakázáno
- Nikdy nepoužívej `sudo`
- Nikdy necommituj `.env` soubory
- Nepřidávej závislosti bez schválení

## Git workflow
- Větve: `feature/`, `fix/`, `chore/`
- Commit messages v češtině
- Před pushnutím vždy spusť testy

Tipy pro psaní CLAUDE.md

  • Pište stručně – Claude čte celý soubor při každém sezení
  • Zahrňte konkrétní příkazy (ne jen „spusť testy")
  • Uveďte co Claude nesmí dělat (zakázané akce)
  • Popište strukturu adresářů pro velké projekty
  • Commitujte CLAUDE.md do gitu – je součástí projektu

Kompletní příklad

{

Hooks

Shell skripty automaticky spouštěné při specifických událostech

ℹ️

Hooks jsou shell příkazy v settings.json, které se automaticky spustí při určitých událostech – například po každém zápisu souboru, nebo po každém shell příkazu. Harness (ne Claude) je spouští.

Typy hooků

TypKdy se spustí
PreToolUsePřed každým voláním nástroje (může ho zablokovat)
PostToolUsePo každém volání nástroje (pouze informativní)
StopKdyž Claude dokončí svou odpověď
NotificationPři systémových notifikacích

Příklady hooků

Auto-formátování po každém zápisu

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "ruff format $CLAUDE_TOOL_INPUT_FILE_PATH 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

Notifikace po dokončení úkolu

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "notify-send 'Claude Code' 'Úkol dokončen!' 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

Bezpečnostní kontrola před shell příkazy

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'rm -rf' && exit 2 || exit 0"
          }
        ]
      }
    ]
  }
}

Proměnné prostředí v hoocích

ProměnnáHodnota
$CLAUDE_TOOL_NAMENázev volaného nástroje (Bash, Write, …)
$CLAUDE_TOOL_INPUTJSON vstup nástroje
$CLAUDE_TOOL_INPUT_FILE_PATHCesta k souboru (pro Write/Edit)
$CLAUDE_TOOL_OUTPUTVýstup nástroje (PostToolUse)
⚠️

Hook s exit kódem 2 zablokuje volání nástroje (PreToolUse). Jiné exit kódy jsou ignorovány (PostToolUse je vždy pouze informativní).

Agent Tool

Spouštění subagentů s vlastním kontextem, sadou nástrojů a specializací

ℹ️

Agent tool umožňuje hlavnímu Claudovi (orchestrátoru) spustit jednoho nebo více subagentů. Každý subagent dostane vlastní prompt, sadu nástrojů a pracovní prostor – a vrátí výsledek zpět orchestrátoru.

Základní princip

# Orchestrátor pošle subagentovi úkol:
Agent({
  "description": "Analýza bezpečnosti API endpointů",
  "prompt": "Projdi všechny soubory v src/api/ a najdi
    potenciální bezpečnostní zranitelnosti. Zkontroluj:
    SQL injection, XSS, autentizaci. Vrať seznam problémů.",
  "subagent_type": "general-purpose"
})

Typy subagentů

TypSpecializaceDostupné nástroje
general-purpose Obecný agent pro komplexní úkoly Všechny nástroje
Explore Rychlé read-only prohledávání kódu Read, Grep, LS, Glob, WebSearch
Plan Návrh implementačních plánů Read-only + ExitPlanMode
claude-code-guide Dotazy na Claude Code samotný Bash, Read, WebFetch, WebSearch

Parametry Agent toolu

ParametrTypPopis
descriptionstring povinnýKrátký popis úkolu (3–5 slov)
promptstring povinnýKompletní instrukce pro subagenta
subagent_typestringTyp specializovaného agenta
modelstringPřepsání modelu (sonnet/opus/haiku)
run_in_backgroundbooleanSpustit na pozadí, nekoukat na výsledek hned
isolationstring"worktree" – izolovaná kopie repozitáře

Kdy použít Agent tool

  • Velké codebase průzkumy – prohledávání stovek souborů
  • Izolované úkoly – subagent s jiným zaměřením (bezpečnost, testy, dokumentace)
  • Ochrana kontextu – výsledky subagenta se vrátí shrnuté, ne jako surový výstup
  • Paralelní práce – více agentů pracuje současně na různých částech

Izolace pomocí worktree

Agent({
  "description": "Riziková refaktorizace",
  "prompt": "Přepište modul auth.py na nový systém tokenů.",
  "isolation": "worktree"
  // Subagent pracuje na izolované kopii repozitáře.
  // Změny se nezaplní do vaší větve automaticky.
})
💡

Worktree isolace je skvělá pro experimenty: subagent provede změny v dočasné větvi. Pokud je výsledek špatný, změny se zahodí. Pokud dobrý, můžete je mergnout.

Nastavení agentů

Jak konfigurovat agenty – model, nástroje, kontext, oprávnění a výstup

Přehled všech parametrů

Agent({
  // ─── Povinné ────────────────────────────────────────────
  "description": "Krátký popis (3–5 slov)",   // pro přehled v logu
  "prompt": "Kompletní instrukce...",        // soběstačný prompt

  // ─── Výběr agenta ───────────────────────────────────────
  "subagent_type": "general-purpose",    // typ specializace

  // ─── Model ──────────────────────────────────────────────
  "model": "sonnet",                      // sonnet | opus | haiku

  // ─── Izolace ────────────────────────────────────────────
  "isolation": "worktree",               // izolovaná git větev

  // ─── Spuštění ───────────────────────────────────────────
  "run_in_background": false              // true = spustit na pozadí
})

Výběr modelu pro agenta

Každý agent může mít jiný model. Tím lze optimalizovat cenu i rychlost celého systému:

HodnotaModelKdy použít pro agenta
"opus" claude-opus-4-7 Orchestrátor, složité plánování, kritické rozhodnutí
"sonnet" claude-sonnet-4-6 Implementace kódu, review, většina workerů
"haiku" claude-haiku-4-5 Průzkum kódu, jednoduché transformace, rychlé úkoly 
# Orchestrátor (Opus) řídí levnější workery:

# Průzkumné agenty – rychlé a levné
Agent({
  "description": "Průzkum kódu",
  "prompt": "Najdi všechny funkce bez typových anotací",
  "subagent_type": "Explore",
  "model": "haiku"   // nejrychlejší a nejlevnější
})

# Implementační agent – potřebuje víc inteligence
Agent({
  "description": "Doplnění type hints",
  "prompt": "Doplň type annotations do všech funkcí v src/",
  "model": "sonnet"  // dobrý poměr cena/výkon
})

Nastavení dostupných nástrojů

Typ agenta (subagent_type) určuje, které nástroje má k dispozici:

TypDostupné nástrojeTypické použití
general-purpose Všechny (Bash, Read, Write, Edit, Web…) Implementace, opravy, obecné úkoly
Explore Read, Grep, LS, Glob, WebSearch, WebFetch Průzkum kódu (nemodifikuje soubory)
Plan Vše kromě Edit/Write/Bash destructive Návrh řešení, architektura
claude-code-guide Bash, Read, WebFetch, WebSearch Dotazy na Claude Code funkce
💡

Použijte Explore vždy, když agent má pouze číst a analyzovat – nikoli psát. Je bezpečnější (nemůže nic pokazit) a rychlejší, protože nepotřebuje schvalování oprávnění.

Worktree izolace – nastavení

S "isolation": "worktree" agent dostane izolovanou kopii repozitáře v nové větvi:

result = Agent({
  "description": "Experimentální refaktoring",
  "prompt": "Přepiš celý modul billing/ na async/await",
  "isolation": "worktree"
})

# result obsahuje:
#   - path: cesta k dočasnému worktree
#   - branch: název vytvořené větve
# Pokud agent neprovede žádnou změnu, worktree se smaže automaticky.

Jak napsat dobrý prompt pro agenta

⚠️

Agent nevidí historii konverzace s orchestrátorem. Veškerý potřebný kontext musí být obsažen přímo v poli prompt.

Struktura dobrého promptu pro agenta:

Agent({
  "description": "Oprava JWT validace",
  "prompt": """
    KONTEXT:
    Pracuješ v repozitáři Flask REST API.
    Soubor s bugem: src/auth/tokens.py, funkce verify_token().

    PROBLÉM:
    Funkce vrací True i pro expirované JWT tokeny.
    Pole 'exp' v payload není kontrolováno.

    ÚKOL:
    1. Přečti soubor src/auth/tokens.py
    2. Oprav verify_token() – přidej kontrolu time.time() > payload['exp']
    3. Spusť testy: pytest tests/test_auth.py -v
    4. Pokud testy projdou, vytvoř commit s popisem opravy

    VÝSTUP:
    Vrať: 'OPRAVENO' nebo popis problému, pokud se nepodařilo opravit.
  """
})

Paralelní agenti

Spouštění více agentů současně pro masivní zrychlení komplexních úkolů

⚡

Klíčové pravidlo: Pokud jsou dva úkoly na sobě nezávislé, vždy je pošlete agentům v jediné zprávě jako dvě volání Agent toolu. Tím se spustí souběžně.

Jak funguje paralelismus

Když orchestrátor (hlavní Claude) zavolá několik Agent toolů v jedné odpovědi, harness je spustí souběžně. Orchestrátor čeká na všechny výsledky a pak je zpracuje dohromady.

# Orchestrátor volá 3 agenty NAJEDNOU (ve stejné odpovědi):

# Agent 1 – analýza backendu
Agent({
  "description": "Backend code review",
  "prompt": "Analyzuj src/backend/ – najdi performance problémy",
  "subagent_type": "Explore"
})

# Agent 2 – analýza frontendu (souběžně!)
Agent({
  "description": "Frontend code review",
  "prompt": "Analyzuj src/frontend/ – najdi performance problémy",
  "subagent_type": "Explore"
})

# Agent 3 – testy (souběžně!)
Agent({
  "description": "Test coverage analysis",
  "prompt": "Zjisti pokrytí testy a navrhni chybějící testy",
  "subagent_type": "Explore"
})

Sekvenční vs. paralelní

SituacePřístupProč
Agenti pracují na nezávislých modulech Paralelně Žádná závislost, plný souběh
Agent B potřebuje výsledek agenta A Sekvenčně Závislost na výstupu
Psaní do různých souborů Paralelně Žádný konflikt
Psaní do stejného souboru Sekvenčně Race condition
Spouštění testů různých modulů Paralelně Nezávislé sady testů

Pozadí (background) agenti

Použijte run_in_background: true pro agenty, jejichž výsledek nepotřebujete okamžitě:

Agent({
  "description": "Generování dokumentace",
  "prompt": "Vygeneruj API dokumentaci pro všechny endpointy
    v src/api/ a ulož ji do docs/api.md",
  "run_in_background": true
  // Orchestrátor hned pokračuje s jinými úkoly.
  // Notifikace přijde, jakmile agent skončí.
})

Spusť souběžně:

# KROK 2 (po K1): Orchestrátor zpracuje výsledky a rozhodne

Spusť souběžně:
  - Agent D: "Vytvoř migrations/0042_add_indexes.py"

  - Agent F: "Aktualizuj tests/test_db.py o nové testy"

Multi-agent architektury

Vzory pro koordinaci více agentů při komplexních softwarových úkolech

Vzor 1: Hub & Spoke (Hvězda)

Jeden orchestrátor koordinuje více specializovaných agentů. Nejjednodušší a nejčastější vzor.

┌─────────────────────────────────────────┐
│            ORCHESTRÁTOR                  │
│      (hlavní Claude, plánuje)            │
└──────────┬──────────┬──────────┬────────┘
           │          │          │
    ┌──────▼──┐ ┌─────▼──┐ ┌────▼──────┐
    │ Agent A  │ │Agent B │ │  Agent C  │
    │  Backend │ │Frontend│ │   Testy   │
    └──────────┘ └────────┘ └───────────┘

# Implementace:
Agent({ "description": "Backend agent", ... })   # paralelně
Agent({ "description": "Frontend agent", ... })  # paralelně
Agent({ "description": "Test agent", ... })     # paralelně

Vzor 2: Pipeline (Řetězec)

Výstup jednoho agenta je vstupem dalšího. Každý agent přidá hodnotu.

Požadavek → [Agent A: Analýza] → [Agent B: Implementace]
          → [Agent C: Review] → [Agent D: Testy] → Výsledek

# Každý krok čeká na předchozí:
analysis = Agent({ "prompt": "Analyzuj požadavek a navrhni řešení" })
impl     = Agent({ "prompt": f"Implementuj: {analysis}" })
review   = Agent({ "prompt": f"Zkontroluj implementaci: {impl}" })
tests    = Agent({ "prompt": f"Napiš testy pro: {impl}" })

Vzor 3: Map-Reduce

Problém se rozdělí na části (map), paralelně zpracuje a výsledky se sloučí (reduce).

# MAP fáze – paralelní zpracování 4 modulů najednou:
results = [
  Agent({ "prompt": "Analyzuj modul auth" }),    # paralelně
  Agent({ "prompt": "Analyzuj modul billing" }), # paralelně
  Agent({ "prompt": "Analyzuj modul users" }),   # paralelně
  Agent({ "prompt": "Analyzuj modul api" }),     # paralelně
]

# REDUCE fáze – orchestrátor sloučí výsledky:
Agent({ "prompt": f"Slouč analýzy a vytvoř souhrnnou zprávu: {results}" })

Vzor 4: Hierarchická síť

Orchestrátor deleguje na sub-orchestrátory, kteří řídí vlastní tým agentů.

        HLAVNÍ ORCHESTRÁTOR
        /          |          \
   Feature A    Feature B    Feature C
  (sub-orch.)  (sub-orch.)  (sub-orch.)
   /    \       /    \       /    \
  A1    A2     B1    B2     C1    C2
(workers)    (workers)    (workers)

Vzor 5: Spectre (Specialisté + Generalisté)

RoleSpecializaceTypické úkoly
OrchestrátorPlánování a koordinaceDekompozice problému, sbírání výsledků
Explore agentRead-only průzkum kóduHledání souborů, analýza, grep
Plan agentArchitektura a plánováníNávrh řešení, identifikace rizik
Worker agentImplementacePsaní kódu, testy, dokumentace

Komunikace mezi agenty

⚠️

Agenti nesdílejí kontext – každý subagent dostane pouze to, co mu orchestrátor explicitně předá v promptu. Navrhujte prompty tak, aby byly soběstačné: obsahovaly vše, co agent potřebuje.

# Špatně – agent nemá kontext:
Agent({ "prompt": "Oprav to" })

# Dobře – agent dostane veškerý potřebný kontext:
Agent({
  "description": "Oprava auth bug",
  "prompt": """
    Oprav bug v souboru src/auth.py, funkci verify_token().
    Bug: funkce vrací True i pro expirované tokeny.
    Příčina: chybí kontrola pole 'exp' v JWT payload.
    Požadavek: přidej kontrolu exp > time.time().
    Testy jsou v tests/test_auth.py.
    Spusť testy po opravě: pytest tests/test_auth.py -v
  """
})

Supervisor vzory

Orchestrátor jako inteligentní koordinátor, kontrolor kvality a plánovač

ℹ️

Supervisor (orchestrátor) je agent zodpovědný za celkový výsledek. Dělí práci, deleguje ji, sbírá výsledky, ověřuje kvalitu a opravuje chyby. Jednotliví workeři (subagenti) nevědí nic o ostatních – vědí jen svůj úkol.

Vzor 1: Supervisor s kontrolou kvality

Po každé dávce workerů orchestrátor zkontroluje výsledky před dalším krokem.

# 1. Deleguj práci
result_a = Agent({ "prompt": "Implementuj přihlašovací formulář" })
result_b = Agent({ "prompt": "Implementuj registrační formulář" })

# 2. Zkontroluj výsledky (orchestrátor rozhoduje)
if problems_found(result_a):
    # 3. Pošli opravu zpět
    fix = Agent({ "prompt": f"Oprav tyto problémy: {problems}" })

# 4. Pouze po úspěšné kontrole pokračuj
Agent({ "prompt": "Napiš testy pro oba formuláře" })

Vzor 2: Critic-Reviser (Recenzent + Opravář)

Jeden agent vytvoří, druhý zkritizuje, třetí opraví.

# Fáze 1: Vytvoření
draft = Agent({
  "description": "Návrh architektury",
  "prompt": "Navrhni architekturu pro platební systém"
})

# Fáze 2: Nezávislá kritika
critique = Agent({
  "description": "Bezpečnostní review",
  "prompt": f"""
    Zkritizuj tento návrh architektury z pohledu:
    1. Bezpečnosti (OWASP Top 10)
    2. Škálovatelnosti
    3. Chybějících komponent
    Návrh: {draft}
  """
})

# Fáze 3: Revize na základě kritiky
final = Agent({
  "description": "Finální architektura",
  "prompt": f"Vylepši návrh {draft} na základě kritiky: {critique}"
})

Vzor 3: Tournament (Turnaj)

Více agentů navrhne řešení, orchestrátor vybere nejlepší.

# Paralelně – 3 různé přístupy k implementaci
solution_a = Agent({ "prompt": "Implementuj cache jako Redis hash" })
solution_b = Agent({ "prompt": "Implementuj cache jako LRU v paměti" })
solution_c = Agent({ "prompt": "Implementuj cache jako SQLite" })

# Orchestrátor porovná a vybere
winner = Agent({
  "prompt": f"""
    Porovnej tato tři řešení cache:
    A: {solution_a}
    B: {solution_b}
    C: {solution_c}

    Kritéria: latence, paměťová náročnost, složitost.
    Doporuč nejlepší a zdůvodni proč.
  """
})

Vzor 4: Self-healing Supervisor

Supervisor detekuje selhání a automaticky zotavuje systém.

# 1. Worker spustí příkaz
result = Agent({ "prompt": "Spusť testy a vrať výsledek" })

# 2. Supervisor vyhodnotí
if "FAILED" in result:
    # 3. Diagnostický agent zjistí příčinu
    diagnosis = Agent({
      "prompt": f"Analyzuj selhání testu a najdi příčinu: {result}"
    })

    # 4. Opravný agent provede fix
    fix = Agent({
      "prompt": f"Oprav problém: {diagnosis}"
    })

    # 5. Verifikační agent ověří opravu
    verify = Agent({
      "prompt": "Spusť testy znovu a potvrď, že jsou zelené"
    })

Vzor 5: Scrum Board

Orchestrátor hraje roli Scrum mastera – vytvoří backlog, přidělí úkoly, sleduje progress.

# Sprint plánování:
tasks = Agent({
  "prompt": "Rozděl user story 'platební brána' na atomické úkoly"
})

# Paralelní sprint – každý worker dostane jeden task:
for task in tasks:
    Agent({
      "description": task.title,
      "prompt": task.description,
      "run_in_background": true
    })

# Sprint review:
Agent({
  "prompt": "Zkontroluj všechny dokončené tasky a vytvoř sprint report"
})

Doporučení pro dobré supervisory

💡
  • Atomické úkoly – každý worker dostane jeden jasný, malý úkol
  • Explicitní výstup – definujte formát výstupu (JSON, Markdown, kód)
  • Nezávislost – workery by neměly záviset jeden na druhém tam, kde to není nutné
  • Checkpointy – supervisor kontroluje po každé fázi, ne jen na konci
  • Fallback – plánujte co dělat, když worker selže

MCP Servery

Model Context Protocol – rozšíření nástrojů Claudea o vlastní zdroje dat a akce

ℹ️

MCP (Model Context Protocol) je otevřený protokol pro připojení AI modelů k externím zdrojům. MCP server vystavuje nástroje (tools), zdroje (resources) a výzvy (prompts), které Claude může používat.

Přidání MCP serveru

Přes příkazovou řádku

# Přidat globálně
$ claude mcp add muj-server npx -y @mcp/muj-server

# Přidat pro projekt
$ claude mcp add --scope project muj-server npx -y @mcp/muj-server

Ručně v settings.json

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "your-token"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres",
               "postgresql://localhost/mydb"]
    }
  }
}

Vytvoření vlastního MCP serveru (Python)

from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent

app = Server("muj-server")

@app.list_tools()
async def list_tools():
    return [Tool(
        name="nacti_konfiguraci",
        description="Přečte konfiguraci aplikace",
        inputSchema={"type": "object", "properties": {}}
    )]

@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "nacti_konfiguraci":
        return [TextContent(type="text", text=read_config())]

if __name__ == "__main__":
    import asyncio
    asyncio.run(stdio_server(app))

Tipy a triky

Best practices pro maximální produktivitu s Claude Code

Produktivita

Spouštějte Claude v kořeni projektu

Claude automaticky nahraje CLAUDE.md, přečte git historii a pochopí kontext. Vždy spouštějte claude z kořenového adresáře projektu, ne ze zanořeného adresáře.

Používejte /compact aktivně

Nečekejte na varování o plném kontextu. Po dokončení větší funkce spusťte /compact s instrukcí, co zachovat:

Vy: /compact Zachovej kontext auth modulu a aktuální stav migrace databáze.

Pipe místo kopírování

# Analýza logu bez kopírování:
$ tail -100 /var/log/app.log | claude -p "najdi anomálie"

# Review diff:
$ git diff | claude -p "zkontroluj bezpečnostní problémy"

# Analýza výstupu testů:
$ pytest --tb=long 2>&1 | claude -p "vysvětli selhání testů"

Bezpečnost

🔴

Nikdy neposkytujte Claudovi API klíče, hesla ani tajné tokeny přímo v textu. Používejte proměnné prostředí přes env v settings.json nebo načítejte ze souborů.

Princip nejmenšího oprávnění

Povolujte jen příkazy, které skutečně potřebujete:

# Dobré – specifická povolení:
"allow": ["Bash(npm test)", "Bash(npm run lint)"]

# Rizikové – příliš obecné:
"allow": ["Bash(*)"]

Kvalita kódu

Poproste o review před implementací

Vy: Plánuji přidat platební bránu Stripe. Jaký je tvůj navrhovaný
     přístup a jaká jsou rizika? Zatím nic neimplementuj.

Iterativní přístup

Raději více malých kroků než jeden velký. Claude snáze ověří a opraví menší změny.
